Problem description

Routing problems in operational research are typically classified based on the location of the service. When the service is performed at nodes, it falls under the vehicle routing problem (VRP) [Braekers et al., 2016]; when performed on edges (or arcs), it is categorized as an arc routing problem (ARP) [Corberán et al., 2021]. The routing problem for Dengue control incorporates elements from both VRP and ARP. On one hand, the service performed by spraying vehicles occurs on the arcs surrounding each block, similar to an ARP. On the other hand, each city block can be conceptualized as a super node, aligning it with VRP characteristics. Given these unique features, we introduce a new problem, the City Block Routing Problem (CBRP).

The CBRP focuses on optimizing the service of city blocks within an urban street network. Vehicles are assigned to service these blocks, but the number of blocks that can be serviced is limited by available resources. A block is considered serviced when a vehicle completely encircles it without taking any detours, and each serviced block yields a specific benefit. The primary objective of the CBRP is to identify the optimal subset of blocks that maximizes overall benefits while determining the best route to service the selected blocks within the given resource constraints.

The insecticide application is more effective when performed at dawn, and right before sunset, periods in which occurs a thermal inversion keeping the insecticide in lower altitudes, where the mosquitoes are usually found [Brazilian Department of Health, 2009]. Once the spraying vehicle starts servicing a given street block, the vehicle must subsequently cover all edges in a clockwise direction before moving to another block. A clockwise traversal of a street block guarantees that the fog of insecticide creates a barrier that prevents mosquitoes from escaping.

Instance Generator

We adopted the maps available in the project Open Street Maps (OSM) [Haklay and Weber], which is a collaborative mapping project to create a free and editable map of the word. We have collected actual Dengue notified cases from 2015 to 2021 from the city health departments of Alto Santo and Limoeiro do Norte. The data was retrieved from the databases of the Information System for Notifiable Diseases (SINAN) [Laguardia et al. (2004)]. From Alto Santo, 582 cases were reported from 2016 to 2021, while Limoeiro do Norte reported 3829 Dengue and 414 Chikungunya cases from 2015 to 2021. The data from both cities give 4825 cases reported in the SINAN. The benchmark of instances used in the computation experiments comprises 39 scenarios based on notified cases from the cities of Alto Santo and Limoeiro do Norte. The address of each notification was applied to the OSMnx library [Boeing, 2017] to obtain their latitude and longitude, mapping them to the street blocks.

Besides instances based on real notifications, we also generate sets of instances based on real dengue outbreak locations in the city of Campinas, São Paulo (Campinas City Hall, 2021), however by using synthetic notification numbers. The synthetic instances were generated varying three attributes: number of neighborhoods; block selection strategy; and distribution of notifications among blocks. or each instance set 80 instances were generated, in a progression of 5 blocks, that is, 5 instances of 5 street blocks, 5 instances of 10 blocks, and so on, up to 5 instances of 80 blocks. In the end, we generated eight sets of instances, totaling 640 instances.

Instances

The instances are organized in compressed files, the structure of folders and files is presented bellow.

  • Folder: Washington-\(S\)
    • Folder: \(V\)
      • washington-\(S\)-\(V\)-\(D\).txt
      • param-washington-\(S\)-\(V\)-\(D\).txt

Where \(S\) is the excerpt size, \(V\) is the number os nodes in the graph and \(D\) the number of terminals.

Each instance consists of a graph and a parameter file in the following formats:

File Format: Graph instance
The informations are inside of SECTIONS:
  • SECTION Comment: contain information about the instance
  • SECTION Graph: containt the number of Nodes \(V\), Edges \(E\), and \(E\) lines from each edge in the format
    • <E \(u\) \(v\) λ ξ ω μ>
    • These values are, respectively, the identifier of edge, the source and target nodes, delay, jitter, bandwidth, and link duration estimate
  • SECTION Terminals: contais the Root and Terminal nodes

File Format: Parameter instance

This file contains four lines containing, respectively, the delay limit, jitter limit, delay variation limit, and bandwidth limit.

File Format: Instance preprocessing
The compressed file contains the results of all the preprocessings presented in Ribeiro et. al. [2]* for all our set of instances. Each line contains the identifier of the procedure and the node/edge that can be removed from the graph.

Results

The results were obtained with an implementation in C++ and compiled with GCC version 7.5.0. We solved the ILP models with Gurobi version 9.1. To solve shortest paths and shortest path with resource constraints, we employed the Boost C++ Library, version 1.75. Below you can find the machine specification for the set of instances.

Instances CPU #cores Clock Cache RAM
Washington-MS-MRP-QoS Intel(R) Xeon(R) E5-2630 v4 10 2.2 GHz 25 MB 64 GB

The results are in csv files compressed by algorithm and configuration, each one corresponding to the set of instances. Each row of each file corresponds do the result of one instance. The formulation results contains columns indicating the name of the instance, number of terminals, number of edges, lower/upper bounds, gaps, number of nodes of the Gurobi tree, and the computation time used to find such bounds. The Lagrangain relaxations contains columns indicating the name of the instance,number of terminals, edges, first, best, and iteration of best lower/upper bounds, gaps, and the computation time used.
